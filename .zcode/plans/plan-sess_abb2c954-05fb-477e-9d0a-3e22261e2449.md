# 性能优化 + API/SQL 完善

## 一、性能优化（3 项）

### P1. GROUP BY 快路径使用 HashMap 替代线性扫描（31ms → ~15ms）
- **文件**: `src/sql/executor.rs:2943-2952`
- **问题**: GROUP BY 快路径对每行做 `group_keys.iter().enumerate()` 线性扫描（O(rows×groups)），但同函数已分配了 `key_index: HashMap` 却未在快路径使用
- **修复**: 快路径用 `key_index.get(s)` 替代线性扫描，首次出现用 `key_index.insert()`
- **预期**: 2M行×5组从 10M 次字符串比较 → 2M 次 hash lookup，约 2x 加速

### P2. ORDER BY LIMIT 用 select_nth_unstable 替代 BinaryHeap（28ms → ~18ms）
- **文件**: `src/storage/col_segment/store.rs:2365-2375`
- **问题**: top_k 对每行做 heap push（O(N log K)），可以用 `select_nth_unstable` 做 O(N) 选择
- **修复**: 快路径收集 (ord_key, row_idx) 到 Vec，然后 `select_nth_unstable(k)` + sort 前 K 个
- **预期**: 减少堆操作开销，约 1.5x 加速

### P3. Integer GROUP BY 快路径 + MIN/MAX 正确性修复
- **文件**: `src/sql/executor.rs:2818`
- **问题**: `GROUP BY category_id`（整数列）跳过快路径走慢的 materialized 路径；MIN/MAX 在某些路径返回 NULL
- **修复**: 扩展 col_segment_group_by 支持 Integer 分组列

## 二、SQL 功能完善（4 项）

### S1. CASE WHEN 表达式
- **文件**: `src/sql/ast.rs`（新增 `Expr::Case` 变体）, `src/sql/parser.rs`（解析 CASE WHEN...THEN...ELSE...END）, `src/sql/evaluator.rs`（求值）
- **用途**: 条件表达式，报表/ETL 必备

### S2. UNION / UNION ALL
- **文件**: `src/sql/ast.rs`（新增 `Statement::SetOp`）, `src/sql/parser.rs`（解析 UNION）, `src/sql/executor.rs`（执行：两个 SELECT 结果合并）
- **用途**: 合并查询结果集

### S3. STDDEV / VARIANCE 聚合函数
- **文件**: `src/sql/evaluator.rs`（新增匹配）, `src/sql/executor.rs`（聚合计算路径）, `src/storage/col_segment/store.rs`（扩展 AggregateResult）
- **用途**: 统计分析，性能监控数据

### S4. ALTER TABLE ADD COLUMN
- **文件**: `src/sql/parser.rs`（解析 ALTER TABLE t ADD COLUMN）, `src/database/table.rs`（执行 schema 变更）
- **用途**: schema 演进，无需 DROP+ recreate
