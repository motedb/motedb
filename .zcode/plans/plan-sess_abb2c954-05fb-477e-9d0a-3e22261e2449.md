# 性能 +30%，峰值 RSS ≤100MB，接口 p99 <15ms

## 当前状态（commit 3e88a69, 2M rows）

| 指标 | 当前值 | 目标 | 差距 |
|------|--------|------|------|
| Peak RSS | 79MB | ≤100MB | ✅ 已达标 |
| Point query p99 | 9µs | <15ms | ✅ 已达标 |
| AGG (SUM/AVG/MIN/MAX) | 31ms | 需提升 | +30%→22ms |
| GROUP BY | 32ms | 需提升 | +30%→22ms |
| LIKE prefix | 26ms | 需提升 | +30%→18ms |

重点：扫描查询（AGG/GROUP BY/LIKE）的 p99 远超 15ms，需要优化到接近 15ms。

## 3 项优化（按 ROI 排序）

### O1. AGG 列主序原始切片迭代（预计 +40-50%，31→16ms）
- **文件**: `src/sql/executor.rs` 多列无 WHERE 聚合快路径（约 2355-2396 行）
- **问题**: 当前 `for i in 0..n { for ai { fs.get_i64(i) } }` 逐行调用 get_i64 → SegData::slice match 开销，不向量化
- **修复**: 改为列主序 `for ai { let raw = fs.raw_i64_slice(); for &v in raw { sum += v } }`，直接迭代 typed slice，自动 SIMD 向量化
- **参照**: `store.rs:1895-1935` 的 aggregate_filtered 已有同样的优化模式

### O2. LIKE prefix count-only 路径（预计 +30-40%，26→16ms）
- **文件**: `src/storage/col_segment/store.rs` scan_row_indices_prefix + `src/sql/executor.rs` LIKE 分支
- **问题**: COUNT(*) WHERE LIKE 分配 Vec<(seg,row)> 最多 65536 条目仅用于取 .len()；load_full_keys 无条件调用
- **修复**: 添加 `count_prefix_matches` 方法（计数而非 push），COUNT(*) 走此路径；跳过 load_full_keys 当单 segment

### O3. GROUP BY 索引化累加器（预计 +25-35%，32→22ms）
- **文件**: `src/sql/executor.rs` col_segment_group_by（约 2736-2792 行）
- **问题**: Vec<(Box<str>, i64)> 线性扫描做字符串比较（5 组 × 2M 行 = 10M 次 string compare）
- **修复**: 用 HashMap<&str, usize> 映射到 Vec<i64> counts 数组，热路径变为 hash lookup + i64 increment
