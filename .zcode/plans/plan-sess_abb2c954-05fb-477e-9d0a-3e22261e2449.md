# 性能提升 30% — 跨查询列解码缓存

## 核心问题
每次扫描查询都重新解码列数据（read_fixed_i64/read_text → 分配 Vec + 解压）。基准测试跑同一查询 10 次，每次都重新解码。已有的 `col_cache` 只用于 point query，扫描路径绕过了它。

## 方案

### 1. 扫描路径使用 col_cache（最大收益）
- **文件**: `src/storage/col_segment/segment.rs`
- 添加 `read_fixed_cached(col_idx) -> Option<FixedSegment>` 和 `read_text_cached(col_idx) -> Option<TextSegment>`
- 这些方法先查 `col_cache`，miss 时解码并缓存
- 已有的 `CachedCol::Fixed/Text` + `BoundedColCache` 基础设施完全可复用

### 2. 聚合快路径使用缓存的方法
- **文件**: `src/sql/executor.rs`
- SUM/AVG/MIN/MAX 快路径：`seg.sst.read_fixed_i64(c)` → `seg.read_fixed_cached(c)`
- GROUP BY 快路径：`seg.sst.read_text(group_pos)` → `seg.read_text_cached(group_pos)` + agg 列同样
- COUNT WHERE：`seg.sst.read_text(filter_col)` → `seg.read_text_cached(filter_col)`
- LIKE prefix：同上

### 3. 聚合路径不清除缓存
- **文件**: `src/sql/executor.rs`
- 聚合路径的 `store.release_query_memory()` 改为不清除 col_cache
- 只有写操作（INSERT/UPDATE/DELETE）清除缓存（已通过 append_rows/flush_buffer 的 clear_cache 实现）

### 4. COL_CACHE_CAP 验证
- 当前 CAP=16，5 列 × 1-2 segment = 5-10 entries，足够

## 预期结果
- 迭代 2-10（基准测试的 9/10 次重复）：列解码从 ~10ms → ~0ms
- AGG: 14ms → ~5ms（+64%）
- GROUP BY: 20ms → ~10ms（+50%）
- LIKE/COUNT: 7-11ms → ~4-6ms（+35%）
- 10 次平均：~30-50% 提升

## 风险
- 稳态 RSS 增加（缓存持有解码数据 ~30-80MB）
- 写操作正确清除缓存（已有机制，低风险）
