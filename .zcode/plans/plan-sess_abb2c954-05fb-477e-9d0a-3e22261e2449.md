# MoteDB 性能优化 + Bug 修复方案

基于对 SQL 执行器(18K行)、存储引擎、API/CRUD 层的全面代码审查,发现 6 个正确性 bug + 5 个高价值性能优化点。所有发现均已通过代码行号验证。

## 第一批:P0 正确性 Bug 修复(6个,每个独立可修)

### B1. NULL 比较语义不一致 (SQL 三值逻辑违反)
- **文件**: `src/sql/executor.rs:8045`
- **问题**: `eval_expr_simple` 对 NULL 比较返回 `Bool(false)`,而 `eval_expr_on_row`(8991)正确返回 `Null`。导致 `WHERE NOT (x < 10)` 在 x=NULL 时错误包含行。
- **修复**: 8045 行 `Ok(Value::Bool(false))` → `Ok(Value::Null)`
- **风险**: 极低(一行改动,行为更正确)

### B2. ORDER BY 两条路径排序不一致
- **文件**: `src/sql/executor.rs:14103-14111`
- **问题**: 老路径用 `partial_cmp().unwrap_or(Equal)` 绕过了 `compare_values`(832行定义的 NULL/NaN 排序),导致 NULL/NaN 在不同 ORDER BY 路径下排序不同。
- **修复**: 14103 改用 `compare_values` 函数
- **风险**: 低

### B3. HAVING 静默吞错
- **文件**: `src/sql/executor.rs:12767`
- **问题**: `.unwrap_or(false)` — HAVING 求值错误被静默当"该组不通过",用户无感知。
- **修复**: 改为 `?` 传播错误,或 `warn_log!` + 跳过
- **风险**: 低

### B4. WAL decode 无限递归 (DoS 漏洞)
- **文件**: `src/txn/wal.rs:428`
- **问题**: `decode_with_fallback` 对嵌套压缩标记无限递归,恶意 WAL 文件可导致栈溢出。
- **修复**: 加递归深度参数,超过1层返回错误
- **风险**: 低

### B5. 显式 AUTO_INCREMENT ID 不检查重复 PK
- **文件**: `src/database/crud.rs:139-166`
- **问题**: 显式指定 ID 时只推进计数器,不检查 PK 是否已存在 → 可插入重复 PK。
- **修复**: 在 explicit_id 路径加 PK 唯一性检查(复用已有的 pk_lookup/query_by_column)
- **风险**: 低(加一个检查)

### B6. WAL group_commit 超时回退导致重复写入
- **文件**: `src/txn/wal.rs:1611-1646`
- **问题**: 超时后内联写入,但队列里的记录副本也会被 group-commit 线程写入 → 重复 WAL 条目。
- **修复**: 超时回退时标记队列记录为已处理(或从队列移除)
- **风险**: 中(需仔细处理并发)

## 第二批:P1 高价值性能优化(5个)

### P1. read_segment_bytes 缓存文件句柄 (最大 I/O 瓶颈)
- **文件**: `src/storage/lsm/columnar.rs:1047-1066`
- **问题**: 大文件(>256KB)每次 read_fixed_i64/read_text 都 `File::open` + seek + read + close。全表扫描 300K 行 × N 列 = 数百次 open/close 系统调用。
- **修复**: 在 ColumnarSSTable 里缓存一个 `Mutex<File>` 句柄,避免重复打开。或启用 mmap(当前 mmap 是死代码)。
- **预期收益**: 大幅减少系统调用,扫描类查询提速

### P2. PK 缓存读路径无锁快速通道
- **文件**: `src/database/pk_cache.rs:80-90`
- **问题**: 每次 PK 查找都要 write lock(LRU touch),并发读时序列化。
- **修复**: 加 read-only 快速路径(先尝试 read lock,命中则跳过 LRU touch)
- **预期收益**: 并发查询吞吐提升

### P3. batch_get O(K²) 修复
- **文件**: `src/storage/lsm/engine.rs:1126`
- **问题**: `remaining_keys.iter().position()` 线性扫描,O(K²)。
- **修复**: 用 HashMap 存位置映射

### P4. 热路径 format! 消除
- **文件**: `src/sql/executor.rs` (141处,多处在逐行循环)
- **问题**: `format!("{}.{}", table, col)` 在行循环里反复构造。
- **修复**: schema 级别预算 key

### P5. unreachable!() 改为显式错误
- **文件**: `src/sql/executor.rs:2574/8052/8998/15649`
- **问题**: 4处 unreachable!() 在输入异常时 panic 整个进程。
- **修复**: 改为返回 Internal 错误

## 实施顺序

1. **B1-B5** (正确性,改动小、风险低、独立可修) — 先做
2. **B6** (WAL 重复写入,需更仔细)
3. **P1** (文件句柄缓存,最大性能收益)
4. **P2-P5** (其他性能优化)
5. 每批修完跑 `cargo test --lib` + bench 验证无回归

## 验证策略

- 每个修复后: `cargo test --lib`(429个)+ 相关集成测试
- P0 全部修完后: `cargo test --workspace`(全量)
- P1 修完后: `bench_vs_sqlite` 对比性能数字
- 全部完成后: fmt + clippy + 重新打 tag 推送
